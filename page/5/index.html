面试官 Tips

有幸听到大佬的对面试的经验，想着写个总结吧。期望有一天自己能用到。

首先是要明确面试的目的- 不要尝试找最优秀的人，要找合适的人，主要是因为找优秀而又合适的人是非常困难的。

合适的定义如下：

  1. 满足此时此刻的需求
  2. 有潜在的未来成长空间

在看一个人面试者的时候，需要从下面三个方面来考量

  1. 思维方式
  2. 知识结构
  3. 背后动机

不要玩技巧，大家都是成年人。那些建立在信息不对等的情况下有自我优越感的，瞧不起别人的人，都是傻逼。面试是一个坦诚沟通的花间。

两个ToDo：

  1. 要尊重别人

    面试官要在有限的时间内让面试者多知道公司的信息，优缺点。

    不要使用压力面试，是真诚的请人来共事的

  2. 保持敬畏之心，因为面试官代表的是公司的脸面

    不要让来面试的人走了之后，有对公司嚼舌根的可能性，应该在面试的时候保持专业和尊重的态度

两个Not ToDo：

  1. 不让人自我介绍
  2. 迟到

事先设计好问题，举例以及注意点如下：

  • 成长最深刻的项目是什么，当时情况是什么，外部挑战，内部挑战，你是什么角色，你怎么处理
  • 事实和行为问题 - 简历上的高光时刻
  • 少问知识，问实战经验
  • 问开放性问题，不要问假设，引导，封闭性问题
  • 不要直接问什么
  • 不要被讲的事实所欺骗，很有可能面试者只是参与了一小部分事件，而不是主导
  • 10分钟一个能力（不是知识）
  • 讲三个，不要超过，也不要少于
  • 尝试了解背后的动机，但是不要被骗

总结一下，一下几个点最为重要：

  • 身体语言
  • 不要压力面试，信息不对称的优越感，盛气凌人
  • 要记录事实，之后可以进行判断
  • 不要从知识出发，而是要看到知识背后的能力
  • 真诚，平等，珍重，是找伙伴，不要设计小陷阱

Fortify学习笔记

  • 前言，思考

    之前做Code Audit的时候，用的是Checkmarx，而且是站在乙方的角度。而最近在新公司，站在甲方的角度用另一个同样著名的软件Fortify去做的时候，让我有了不一样的思考。

    这个不一样不是因为我使用的软件不同了，软件层面上，当两个软件都具有一定的市场竞争力的时候，他们的功能就大同小异了。主要的差别，最大的还是来自甲乙方的责任与态度。

    先说说乙方。在乙方做测试的时候非常简单，你给我代码，无论是在你给我提供的电脑上，还是你把代码整个都给我，无所谓。我需要做的就是把代码跑一遍，然后把所有跑出来的结果再进行一一的验证。这样做一般性都是project based，不存在重复性，所以每次做都是崭新的一次，每次花的时间都是根据代码的总长度而言。所需要的技术一般性就是阅读代码的技术，但是这个往往是不够的，还需要做大量的搜索，甚至于需要开发的帮助。

    然后对甲方而言，情况就大不相同了。首先，甲方这边不是project based的，不是完成一次项目，下一个就完全不一样了。很大的概率上来讲，是存在重复性的。其次，这样的项目一般性都不太可能让开发把代码一个个丢给你，然后你一个个去扫，再看。很大程度上是需要自动化，整合到CICID。经验上来讲，这种直接把代码丢进去的扫描，扫出来的结果会有非常高的误报率。所以做甲方的代码审计，需要在初期建立各种各样的规则来降低误报率，这样自动化扫描才会变得有意义。

    因为Fortify是我之前从来没有用过的软件，所以我花了些时间学习了一下并且做了以下的笔记。

    如何降低误报率

    Fortify降低误报率的办法有两种，分别是使用Cleanse Rules或者使用Audit Template：

    Cleanse Rules

    当在查阅一个扫描结果的时候，如果我们很确定这个结果是一个误报，那么我们可以把这个结果加入Cleanse Rules，达到不报警的目的。这种办法可以细致到某个package，某个class，某个function为止：

    路径： Administration → Advanced Audit Tools → Dataflow Cleanse Rules

    应用场景：

    1. 对被调用的共同库所使用，这样的使用结果可以避免公共库在不同的场景下被调用时，对每一次都发生报警。
    2. 对内部，不重要的代码使用。如果报警对象是内部使用，威胁等级不高以及可以确定逻辑上哪怕没利用的情况下也不会有任何威胁的代码。这样做到目的是有时候修改代码容易导致一系列别的问题，比如downtime以及修改之后别的地方的问题。如果这样的代码存在问题但是几乎是没有影响的话，可以选择加入Cleanse Rules来规避。

    Audit Template

    当查阅到某个扫描结果的时候，如果我们感觉这个结果的分级或者威胁等级不符合，那么我们可以创造一个Audit Template。

    有两种办法来创造Audit Template：

    1. Audit Tools → Add a rule
    2. From the issue → Add audit filter

    相较于Cleanse Rule，这样做更多的目的是在满足某一条件的时候，对目标报警进行更多的操作处理，而不是仅仅是忽略。类似于If-else的逻辑。

    如何把扫描结果自动应用成各个类型

    人工审核每一个扫描的结果是一件非常费时费力的事情。一个应用程式往往会有成百上千的报警。在这种情况下我们可以启用Audit Assistant

    路径： Administration → Variables → Audit Assistant

    首先我们要先启用Audit Assistant, 然后对每个需要进行分类的报警点选auto-apply，这样短时间内我们就可以对所有的报警通过ML分类成下面五大类。根据Fortify自己的介绍，这样分类有着超过99%的准确率，但是实际上手还是要结合公司环境来看。

    如何判断扫描结果的类型

    Credit: Security Code Review Guidelines - https://adam.shostack.org/review.html#code-reliability

    仔细研究了一下代码审计，所有扫出来的问题还需要人工进行以下几个分类，有些比较熟悉，有些却比较陌生，需要研究

    • Not an Issue (False Positive, or other)

      • False Positive，误报，没有什么可以说的

    • Reliability Issue

      相对于安全问题而言，代码存在可靠性的问题，比如没有检查变量类型，没有handle exception之类。这些代码存在潜在性的问题，一旦发生之后有可能造成系统运营商的影响，比如系统瘫痪之类的。

      For Example:

      1. System calls should have their return status checked.
      2. Signals should be caught and handled.
      3. Multithreaded code should use mutex() calls on globals.
      4. C++ classes should have a constructor, destructor, (??)
      5. Configuration information should be in a configuration file, not hardcoded into the program. See the section on configuration files.
      6. The year 2000 is real soon now. Leap years happen every four years. Daylight savings may move the clock back an hour annually.
      7. Functions should perform bounds checking.
      8. umask should be set to a restrictive value.
      9. UIDs should never be -1, which can sometimes happen if you have insufficient type strength, or otherwise try putting UID 65535 in a short.

    • Bad Practice

      每一种语言都有一定自己的规范，violate这些规范就会导致Bad Practice

      • 对于Bad Practice的判断需要一定的开发经验，如果没有相关经验的话，最好做一些搜索来辅助判断
      • 对于特定的编程语言加上bad practice就能搜出一大堆来

    • Suspicious

      如果仅从代码的角度显示已经是可以被exploit的，但是因为无法确定变量所以没有办法100%来确定是否可以exploit

    • Exploitable

      所扫描的代码已经确认是可以被Exploited，十分肯定或者已经通过别的工具攻击成功

WebApp Penetration testing Checklist

前言

今天和老板聊天的时候，老板分享了关于做渗透测试的时候甲方和乙方的态度以及责任。

作为乙方，你所需要提供的就是我这次渗透测试中我发现了漏洞，并且对没有发现的漏洞进行两三页的免责申明，说明。起码我以前的公司是这样的。其实这个也是很常见的，我之前的公司人员参差不齐，有时候你只需要找到一两个问题之后，就可以翘高脚了，反正我报告给得出，老板也不是很懂。

但是作为甲方，那就大大的不同了。如果在甲方你也是只每天翘脚，交一两个漏洞，到时候出事了，那么就真的完蛋了。对比甲方和乙方的不同之处，最大的不同点就是甲方还需要管售后，甚至是售前，而且还是无限期的。（保质到你辞职）

那么自然而然的老板问了我了一个问题，如何确保你自己做的渗透测试做完了，漏洞就不存在了呢？首先我的第一反应就是，其实我做完了测试，我也不能保证没有别的问题了，我只能保证在我的能力范围内已经没有问题了。但是如果这是在甲方的话，这是非常不可取的。你每天努力努力地工作做测试，到时候卷铺盖走人的还是你。

对于这一点，我能给出一个比较简单的近似解。假如现在让你做测试，给你一台电脑，然后装了一堆软件，然后说，那你测吧。多多少少，尤其是在时间紧张的情况下，你很容易漏某些个测试。并不是你不会，而且是你忘记了，这种能力范围内的错误是绝对不允许的。（卷铺盖警告）

因此，我个人的解决办法是对某项特定的测试整理一个checklist，在自己认知的范围内，最大程度上的去做好这个测试。这个checklist里面必须包含所有的关键点，这样能够让你的渗透测试的完整性大大提高。

我本身比较常做的测试是网络应用安全测试以及代码审计，所以我进行了详细的搜索并且结合自己的经验，给出了一个Web App Penetration Test 的 checklist

（以下说的每一步基本上有过做测试经验的人都懂，但是实际操作起来的时候可能就没有那么脑子清楚了。如果有什么不清楚或者没听过的名词，建议学习一下-说实话有些我自己也不是很清楚）

测试通常会有黑盒白盒灰盒，黑盒会包括所有的步骤。所以一下步骤是基于黑盒测试，别的类型的测试适当去除某些步骤就可以啦。

信息收集

  • 通过搜索引擎来进行勘探
  • 对网页服务器的指纹信息进行收集
  • 对网页应用的代码进行调查，查看
  • 查看网页应用的所有资产，例如旁站
  • 如果有登录功能的话，尝试枚举登录有红
  • 查实发现各种终端，因为可能不止有一个
  • 通过各种手段查找admin的登录入口
  • 制造网页报错，尝试从报错中查看一些敏感信息，比如吴福气类型以及所使用的语言
  • 采集目标网站信息
  • 尝试找出网络应用的架构（前段，后端）
  • 尝试找出不同模块直接的联系，交流方式
  • 还有一些别的信息，比如数据流，路径之类

登录系统

  • 测试JWT token
    • 签名认证
      • 有时候none签名可以被接受
      • 弱HMAC keys
      • HMAC vs Public Key Confusion
      • https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/06-Session_Management_Testing/10-Testing_JSON_Web_Tokens
  • 绕过登录
    • 爆破目录
    • SQLi
    • 会话ID的预测
  • 用户名枚举
  • 登录口令通过HTTP传播，进行拦截海阔去
  • 系统的默认登录口令对
  • 注册过程的问题
    • Token通过明文传播
    • 由于输入的字符串过大导致的系统瘫痪 DoS
    • 把XSS payload放入注册用户的每个input field
  • 弱口令爆破
  • 测试密码重置系统
    • 尝试在请求中加入第二个email地址
    • 查看是否有令牌通过HTTP明文传播
    • 预测弱令牌
    • 用户是否被强制重新认证
  • 会话ID在URL里面
  • 登出需要把会话令牌给无效化
  • 检查是否有强制的timeout for session
  • 多次登录后被锁，解锁是否容易简单
  • 可替代的登录方式
    • SSO，Oauth
    • 手机登录
    • Token登录
  • Session Puzzling
    • https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/06-Session_Management_Testing/08-Testing_for_Session_Puzzling
  • 会话劫持
    • https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/06-Session_Management_Testing/09-Testing_for_Session_Hijacking

输入认证

  • XSS
    • Reflected XSS
    • Stored XSS
    • Blind XSS
    • DOM XSS
  • SQLi
  • XXE
  • Command injection
  • SSTI/CSTI
    • Insert ${7*7} into every field you see, if it resolves, investigate further
  • SSRF
    • Add burp collaborator URL in everywhere that URL resolves
  • HTTP parameter polution
    • https://owasp.org/www-project-web-security-testing-guide/latest/4-Web_Application_Security_Testing/07-Input_Validation_Testing/04-Testing_for_HTTP_Parameter_Pollution
  • LDAP injection
  • SSI (server side includes injection)
  • XPath injection
  • Clickjacking
  • GraphQL testing

普遍漏洞利用

  • RFI/LFI
  • 不安全的会话管理
  • 攻击/接管旁站、子站
  • 查看利用用户权限，看看是否可以变成或者给与admin权限
  • 测试cookie的特征
    • 铭感的cookie应该有secure和httpOnly的flag
    • 正确设置域名和路径
    • 有过期的时间
    • Same Site政策
  • CSRF测试
    • 针对一些敏感的功能
  • 文件上传测试
    • 上传恶意文档（比如eicar）
    • 上传限制的绕过
      • 修改上传文件类型（大小写，解析漏洞）
      • 使用null班ytes
  • 从浏览器缓存查看敏感信息
    • 对敏感的网站，查看cache control头部
  • 查看是否存在弱的加密协议
    • 可以使用关键词搜索，MD4,MD5,RC4, Blowfish, SHA-1

以上都是列举的都是技术上的漏洞，但是一个漏洞真正发生的时候，起因很多时候都不是因为技术上存在什么漏洞，而是业务逻辑流程差生了漏洞，这种漏洞的查看还需要要求测试员对产品有一定的了解

业务逻辑缺陷

  • 数据认证和清洁
    • 需要对进来的数据进行清洁，并且与业务内部的进行对比。简单的例子就是一般性数量的地方不可以是负数
    • 在做好本地的验证后，要确保数据也没有办法在服务器端的验证被绕过。本地的验证只要有一定的工具，还是非常容易绕过的。
    • 输入不合理的数据，比如在电话号码的地方输入英文字母，查看是否被接受
  • 测试隐藏的变量并且进行修改，看看是否会有影响。
  • 查看是否有些数据是否只在前段隐藏
  • 查看被disable的地方是否只是在前端被disable（有些网站在你信息输入不完整的时候，提交的按钮是灰色并且不可按的。这个时候可以通过在浏览器修改disable的值来强行让它变得可以按，从而达到绕过的目的）
  • 完整性测试
    • 确认系统能接受那些类型的数据，以及不能够接受那些，然后测试
    • 查看每个用户的权限是否可以修改，读取数据，可以进行越权的测试
    • 在业务逻辑不允许的情况下尝试添加，更改，删除数据记录
  • 测试有时效性的功能
    • 有些功能是有时效性的，比如收到的验证码。查实在过了时效之后还是否可以生效
    • 购物网站的优惠券可能已经过期了，但是这个过期可能只是网站前端的检查，绕过后可能依旧可以使用
  • 查看各项功能是否完整（不一定是安全上scope，甲方可能更加注重这一些，会影响这个品牌的影响力）

中国菜刀，蚁剑，fofa

最近因为工作原因，接触了好多从来听也没有听过的名词。以前考OSCP和OSCE的时候，基本上都是针对Linux系统的操作方法。举个简单的例子，通过HTTP服务之后拿到webshell或者reverseshell，发现目标服务器是linux based服务器，之后我就会很自然的找到一个有读写权限的地方（shm或者etc）然后上传LinEnum.sh来dump目标服务器的信息，从而为提权和长时间控制提供线索。

因为工作原因，最近接触需要接触很多没听说过的东西（I don’t know what I don’t know）。然后再搜索他们的同时经常就会遇到一些词汇，什么直接上菜刀链接，通过fofa找到C段旁站。我就很好奇，这些特么都是什么玩意。

静下心来好好整理了一把，主要有一下三个：（如果以后有更多感兴趣的就再写吧）

  1. 中国菜刀
  2. AntSword蚁剑
  3. fofa

中国菜刀

读很多文章的时候，发现经常有人说什么，上菜刀链接，搞定。问了一下周围学安全的同学，得到的回答菜刀不是违禁品吗？

查了一下，官方的github最后的版本是2016年的，之后貌似没有找到更新的版本。除此之外同一个git资料库下面还有2011和2014的版本。菜刀链接主要是针对php以及asp的网页，理论上讲只要能通过上传一个webshell能进行交互，一般性都没有问题。

我自己的认知里面应该有一下这几种方法：

  1. 通过漏洞上传
  2. 通过解析漏洞上传
  3. 通过PUT这个RESTful API 上传

上传之后，只要在特定位置放入链接，就可以基本上拿到整个网站的拓扑图。

16年之后虽然没有继续更新，但是还有更多的比如CNKnife，中国小刀之类的拓展。因为网上找的很多版本容易有后门，所以我并没有去运行，只是理论上了解了一下。

AntSword蚁剑

Credit: https://www.yuque.com/antswordproject/antsword/kybgpb

之所以不测试中国菜刀是因为我知道蚁剑也有差不多的功能，相比于蚁剑，他本身是一个开源的项目，更加的安全。从官方的GitHub下载后验证md5值，确保没有被修改。

右击可以直接通过wget来下载和上传需要的文件，用于提权或者维持后门。

fofa

简单来说就是个搜索引擎，与一般的引擎不同的是不是基于关键词的搜索。相对的，他是基于某个网络名词进行特定的搜索。比如搜索指定ip，指定域名domain等等。简单来说因为他可以制定目标进行搜索，某种程度上算是一种fuzzing。

其中我觉得最具有攻击性的应该是对指定的域名进行搜索。比如我搜索qq.com，结果中就会出现abc.qq.com或者cde.qq.com。这种子域名可能在正常情况下不访问特定服务是没有办法被发现的。这就是所谓的通过fofa来找C段的旁站。一般情况下，旁站的安全系数会比主站的系统底，以及旁站可能更加容易泄露信息或者攻陷。我们可以把旁站作为一个跳板，然后去攻陷主站。

还有一点就是fofa其实是收费的。。。普通会员最多智能下载一百条信息，但是一般性来说应该就是够用了。

etcd学习笔记

etcd is a strongly consistent, distributed key-value store that provides a reliable way to store data that needs to be accessed by a distributed system or cluster of machines. It gracefully handles leader elections during network partitions and can tolerate machine failure, even in the leader node.

Simple interface

Read and write values using standard HTTP tools, such as curl

Key-value storage

Store data in hierarchically organized directories, as in a standard filesystem

Watch for changes

Watch specific keys or directories for changes and react to changes in values

Features:

  • Optional SSL client certificate authentication
  • Benchmarked at 1000s of writes/s per instance
  • Optional TTLs for keys expiration
  • Properly distributed via Raft protocol

Advantage

  • Fully replicated: Every node in an etcd cluster has access the full data store.
  • Highly available: etcd is designed to have no single point of failure and gracefully tolerate hardware failures and network partitions.
  • Reliably consistent: Every data ‘read’ returns the latest data ‘write’ across all clusters.
  • Fast: etcd has been benchmarked at 10,000 writes per second.
  • Secure: etcd supports automatic Transport Layer Security (TLS) and optional secure socket layer (SSL) client certificate authentication. Because etcd stores vital and highly sensitive configuration data, administrators should implement role-based access controls within the deployment and ensure that team members interacting with etcd are limited to the least-privileged level of access necessary to perform their jobs.
  • Simple: Any application, from simple web apps to highly complex container orchestration engines such as Kubernetes, can read or write data to etcd using standard HTTP/JSON tools.

HTTP 响应拆分（CRLF Injection）

CR - Carriage Return (ASCII 13, \r )

LF - Line Feed (ASCII 10, \n )

Credit to :

https://xz.aliyun.com/t/9707

因为HTTP的header是通过一行行的参数变量以及值的形式来表现，直接去下一行是同一个部分，空一行之后的是HTTP的body

通常比较常见的是HTTP Smuggling， 但是这只是CRLF的一个应用。

CRLF是通过在HTTP的前端植入%0D%0A是的浏览器是翻译的时候会多出一个参数以及内容，从而达到Inject的效果。如果如果在最后在插入一个请求，比如过一个GET， 那么就是传说中的Smuggling了。

通过这个注入我们可以在HTTP的头部中注入我们想要注入的内容，比如Session或者Location。

比较特殊的例子：

1 PHP SoapClinet

该类的构造函数如下：

  • 第一个参数是用来指明是否是wsdl模式，将该值设为null则表示非wsdl模式。
  • 第二个参数为一个数组，如果在wsdl模式下，此参数可选；如果在非wsdl模式下，则必须设置location和uri选项，其中location是要将请求发送到的SOAP服务器的URL，而 uri 是SOAP服务的目标命名空间。

所以可以直接利用第二点，设置null然后我们再进行后面的参数设置。因为后面的参数是可以控制的，

2 Nodejs

因为Node.js使用的是Latin1而不是Unicode，字符会被截断\u0130 就会被截断为 \u30

因此直接是使用%0D%0A是不可行的，要找到四个长度并且以0D和0D结尾的才可以，在latin1中对应的是字符čĊ， 这是比较有趣的一点。

字符 可由以下Unicode编码构造出 Unicode编码对应的字符 Unicode编码对应的字符对应的URL编码
回车符 \r \u010d č %C4%8D
换行符 \n \u010a Ċ %C4%8A
空格 \u0120 Ġ %C4%A0
反斜杠 \ \u0122 Ģ %C4%A2
单引号 ‘ \u0127 ħ %C4%A7

密码找回漏洞小结

Credit to：

https://www.notion.so/29a9518c510c43e8ab9bf8fb937ca5b1#0b0a4988188749cfacae31dc6de09abd

笔者写了很多也很详细，下面的小结只是对我自己的认知的一个反思

  1. 密码重置链接未过期

    简单来说就是可以通过获取这个不会过期链接对密码再次进行修改，有些网站甚至会放上用户名。

  2. 密码重置无速率闲置

    没有设置密码重置的速率导致可以进行重复播放的攻击，实现短信或者邮件的轰炸。会导致资源的浪费以及该功能的瘫痪。

  3. 输入长密码时拒绝服务

    如果没有对长密码进行控制而导致出现错误，一定概率上（这个错误没有被开发考虑到）有可能在错误页面暴露不必要的信息。

  4. 通过密码重置页面进行用户枚举

    这个非常常见，终端可能会对同一账户不同密码进行锁定，但是对不同的账户可能就没有这一个要求。这个时候如果输错密码和输错用户名的提示是不一样的话，这个漏洞就可以被用来枚举用户。最简单的办法就是同一错误提示。

  5. host中毒

    修改host header然后钓鱼受害者，让他把请求送到恶意网站，从而获取修改密码的链接。

  6. refer泄露密码重置令牌

    逻辑不严谨会导致refer西楼密码重置口令

  7. 弱凭证问题

    密码重置链接包含可以轻易被踩到的凭证

  8. 凭证泄露

    可以利用获取的token再进行一次密码重置

  9. 使用电子邮件参数重置密码

    截取修改密码的请求，通过&在收取修改密码链接的邮箱中在加入一个可以控制的邮箱，然后通过收到的链接进行密码修改。

  10. 替换返还包中的信息

    了解包构成的规则，直接把失败的请求修改成功的并且再次请求修改密码，从而绕过验证。但是这种办法的成功率很低

LLMNR和NetBIOS欺骗共计分析

Credit to the following link:

https://www.notion.so/LLMNR-NetBIOS-b97dfeaf9141480b8582cf75b92a3ef6#611cfe466536478ca443d2716f874f73

LLMNT - Link-Local Multicast name resolution

NetBIOS - port 137,138,139

链接当中已经把攻击描述得十分详细，基本原理是一种类似于APR poison的攻击

当DNS服务器不喝医用的时候，终端机器才会调用LLMNT，通过别的子网上的机器的records来进行DNS的获取就，而这个获取的过程是可以被欺骗的

除此之外，在一些特殊场合比如机场，宾馆之类的地方，因为必须链接到他人的网络，所以也是可以加以利用。

主机会先进行本地hosts的查询，如果没有，就会像dns请求。当DNS查询失败或者没有响应，LLMNR才会会被调用，然后主机会通过UDP进行多播，从本地链路上获取所需要的信息。

NetBIOS的作用是将名称解析为IP地址。

检查以确认请求是否针对本地机器名称。 检查最近成功解析的名称的本地缓存。 搜索本地主机文件，该文件是存储在本地计算机上的 IP 地址和名称列表。根据设备的不同，此文件可能已加载到本地缓存中。 查询 DNS 服务器（如果已配置）。 如果启用了 LLMNR，则跨本地子网广播 LLMNR 查询以询问其对等方进行解析。 如果启用了 NetBIOS，如果名称不在本地 NetBIOS 缓存中，则通过向本地子网广播 NetBIOS-NS 查询来尝试 NetBIOS 名称解析。如果如此配置，此步骤可能会使用 Windows Internet 名称服务 (WINS) 服务器以及 LAN 管理器主机 (LMHOSTS) 文件。

攻击方式：

工具：用kali自带的responder.

自己猜测的一种方式：

首先要获得域内的一个账户，什么账户都可以。然后用这账户欺骗受害者（钓鱼）去点击一个不存在的地址，因为这个地址是不存在的，所以会启用LLMNR和NetBIOS。这个时候在同一子网的Responder会欺骗冒充那个不存在的地址，说明这个错误的地址就是自己并且尝试建立SMB链接。受害者主机会把自己的凭证发过去。用户名为明文，然后密码为NTMLv2的hash

这之后只要通过John或者hashcat进行暴力破解就有可能拿到密码。

NetBIOS最常用的三个端口就是137、138、139

137端口：137端口主要用于“NetBIOS Name Service”（NetBIOS名称服务），属于UDP端口，使用者只需要向局域网或互联网上的某台计算机的137端口发送一个请求，就可以获取该计算机的名称、注册用户名，以及是否安装主域控制器、IIS是否正在运行等信息。

138端口：主要提供NetBIOS环境下计算机名的浏览功能。

139端口：通过这个端口进入的连接试图获得NetBIOS/SMB服务。这个协议被用于Windows”文件和打印机共享”和SAMBA。

PORT STATE SERVICE VERSION137/udp open netbios-ns Samba nmbd netbios-ns (workgroup: WORKGROUP)138/udp open filtered netbios-dgm139/tcp open netbios-ssn Microsoft Windows netbios-ssn

防范我觉得笔者写非常好，我就抄写下来了：

防范

1.最实用的方法在每台计算机的 NIC 上禁用 NetBIOS 并通过 DHCP 禁用 LLMNR ，但是这对于一些需要使用到NetBIOS和LLMNR服务的组织不太友好，所以更好的方法是每个系统的主机防火墙上通过阻止 NetBIOS 协议和 TCP 端口 139 以及 LLMNR UDP 端口来限制出站 NetBIOS 和 LLMNR 流量端口为5355。这样就可以有效的防止这几个端口流量出站。

2.但是使用禁用端口流量的方法并不保险，可能攻击者会使用端口转发的方法从另外端口将NetBIOS和LLMNR的流量转发出站。所以另外一种防范办法就是将默认端口改成一些不会引起攻击者注意的高端口，即将端口重定向。这种重定向的方法修改注册表里面的PortNumber修改即可。

[HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\TerminalServer\Wds\rdpwd\Tds\tcp]

[HKEY_LOCAL_MACHINE\\SYSTEM\\CurrentControlSet\\Control\\TerminalServer\\WinStations\\RDP - Tcp ]

3.跟第一点防范方法比较相似的另外一种方法便是给这几个端口配置IPsec安全策略，这样在攻击者试图这几个端口进行扫描的时候就不会得到任何回应。
